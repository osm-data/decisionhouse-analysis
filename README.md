# GPU Allocation Benchmark

Reproducibility artifact for the experiments in:

> **Decisionhouse: Prescriptive Analytics in the Data Stack**
> Matteo Brucato, Fjodor Kholodkov, Soren Little, Jakob Mayer, Duc Nguyen. Under submission at VLDB 2026 (Vision).

## Overview

This benchmark solves a GPU-to-workload allocation problem (the running example from the paper) using three formulations at increasing scale:

1. **Generic MILP** — binary `active` + continuous `quantity` + big-M linking (HiGHS branch-and-bound)
2. **LP relaxation** — continuous variables only, exploiting total unimodularity (HiGHS simplex)
3. **Min-cost flow** — bipartite supply-demand graph (OR-Tools network simplex)

All three formulations produce provably identical optimal solutions. The benchmark measures how structure recognition (selecting the right formulation) affects solve time as problem size grows.

## Requirements

- Python >= 3.13
- [uv](https://docs.astral.sh/uv/) package manager

## Usage

```bash
uv sync                          # install dependencies
uv run benchmark.py              # run all scales (S through 4XL)
uv run benchmark.py --scales S M # run specific scales
uv run benchmark.py --seed 123   # different random seed
uv run plot.py                   # generate figures from results
```

Results are saved to `results/` as JSON (raw data) and PDF/PNG (figures).

## Reproducing paper figures

The formulation scaling figure (Figure 5 in the paper) is generated by:

```bash
uv run benchmark.py --seed 42 --runs 3
uv run plot.py
```

Output: `results/formulation_scaling_gpu.pdf`

## Caching Benchmark

The caching benchmark demonstrates the benefit of model caching and incremental updates for what-if query workloads. It compares two approaches:

1. **Naive** — For each query: read from DuckDB, build full model, solve
2. **Cached** — Build model once, for each query: update RHS only, re-solve

The experiment shows that speedup grows with the number of queries N, as the one-time model build cost is amortized.

```bash
uv run benchmark_caching.py              # Run with XXL scale (500x2000)
uv run benchmark_caching.py --scale 3XL  # Larger scale (1000x4000)
uv run benchmark_caching.py --seed 123   # Different random seed
```
